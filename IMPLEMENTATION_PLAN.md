# iCloud HME Team 接力实施计划

## 1. 目标与性能指标

把同一个 iCloud 账户抽象成一个 HME 资源池，在资源池内长期保护两个 Team 母号，并让每个 Team 空间只轮换一个子号。母号是被动的归属锚点：不登录 Team、不邀请、不退出、不创建令牌；所有 Team 动作都由当前子号和下一子号完成。

闭环目标：

1. 从 `/v2/hme/list` 读取远端 Alias，只接管用户明确选择的 Alias。
2. 被标记为 `team_owner` 的母号永久保护，不进入账号池分配、退役或远端清理路径。
3. 被标记为 `rotating_child` 的子号进入现有账号池，并可作为空间当前号或下一号。
4. 每个 Team 母号分别保存“该母号名下子号的默认 S5/动态链路”；归属该母号的新子号默认继承这条链路，母号本身不使用该链路执行 Team 动作，之后仍可按子号覆盖。
5. 完整工作流成功后，下一号晋升为当前号；旧子号在同一个数据库事务中进入该母号的“已用完”池并永久保留计数。
6. 同一个 iCloud 资源池可以同时服务多个 Team 空间，每个空间绑定一个不同的母号。
7. Team 母号空间平时不预留下一号；用户点击“更换子号”时才现场创建新的 HME Alias、绑定为下一号并启动接力。

稳定性指标：

- 未显式选择的旧 Alias 零写入、零删除。
- `team_owner` 母号进入账号轮换或用完池的概率为零，由数据库约束和服务层双重阻断。
- 重复同步和重复导入均幂等。
- Apple HME 超时或 Session 过期不会阻塞已完成的 Team 轮换；接力完成后不自动修改远端 Alias。
- 敏感 Cookie、IMAP 密码、S5、远端 ID 继续加密保存且不进入日志或前端响应。

## 2. 已有系统分析

- 当前 `icloud_mailboxes` 已能保存一个 iCloud HME Session、转发 IMAP 和资源池 S5。
- 当前 `icloud_aliases` 仅记录本工具新建的 Alias，且每个 Alias 必须对应一个普通账号。
- 当前“检测”只返回远端 Alias 数量，不会同步已有 Alias。
- 当前八阶段工作流已经符合子号接力顺序：旧子号邀请新子号、旧子号退出、新子号登录并确认空间、创建令牌和推送。
- 当前成功提交会把旧账号标记为 `retired`，但不会清理对应的 iCloud Alias。
- 当前模型没有永久母号角色，也没有 Team 空间与母号的绑定关系；母号不能作为工作流执行账号。
- 当前成功提交会从普通可用账号中自动补下一号；iCloud 母号空间必须禁用这条预分配逻辑。

## 3. GitHub 同类项目经验

参考项目：

- `ultra2207/Accounts-Generator-public`：以 `/v2/hme/list` 为事实来源，对 `isActive` 和远端缺失状态做幂等同步。
- `not-knope/Hide-My-Email-iCloud-Manager`：远端清理需要先停用再删除，因此不适合作为接力成功事务的同步依赖。
- `heartmore/icloud-hme`：将 Alias 列表、生成、保留、停用和删除封装为独立 HME 生命周期操作。

采用的结论：

- 远端列表是测量反馈，不凭本地状态猜测 Apple 当前状态。
- 同步与接管分离：同步只展示，接管必须由用户选择。
- 接力成功只更新本地“已用完”状态，不自动停用或删除远端 Alias。
- 本地保留完整账号、母号归属和运行关联，避免审计链断裂并支持累计计数。

## 4. 数据模型设计

数据库升级到 schema v6：

- 重建 `icloud_aliases`，增加角色 `team_owner | rotating_child`，并允许母号没有账号池记录。
- 子号增加 `parent_owner_alias_id`，明确归属两个母号中的一个。
- 母号增加独立加密的“子号默认链路”；iCloud 资源池 S5 只负责 HME 与 IMAP，该默认链路只注入其名下子号的 ChatGPT 工作流。
- 子号增加 `used_at`，接力成功后进入“已用完”池但不删除本地或远端记录。
- `rotating_child` 必须有 `account_id`；`team_owner` 必须没有 `account_id`。
- `workspaces` 增加可空的 `owner_alias_id`，同一母号最多绑定一个空间。
- 现有 v5 Alias 全部无损迁移为 `rotating_child`。

关键约束：

- 母号不能变成可分配账号，不能被状态切换接口退役，不能进入清理队列。
- 空间绑定的母号必须是活动的 `team_owner`。
- 使用 iCloud 子号的空间，其母号和子号必须来自同一个 `icloud_mailbox`。
- 已用完子号不能重新进入可用账号候选，但继续参与母号和空间的累计统计。

## 6. 验收场景

1. 一个 iCloud 返回 20+ Alias，页面默认全部忽略；用户只选择两个母号和两个当前子号后，本地只新增四条受管记录。
2. 两个母号分别绑定两个空间，二者均不进入普通账号池，也不进入任何登录、邀请、退出或令牌阶段。
3. 两个当前子号分别绑定为对应空间当前账号；运行输入只包含当前子号和下一子号，且子号可各自覆盖默认链路。
4. 空间平时没有下一号；点击“更换子号”后才创建一个全新 Alias，继承该母号名下子号默认链路，并立即作为下一子号进入现有八阶段工作流。
5. 工作流失败或取消时，旧子号不进入已用完池；已生成的新子号和 next 绑定保留以便恢复或重试。
6. 工作流完整成功并提交轮换后，旧子号进入对应母号的“已用完”池；母号和远端 Alias 保持不变，母号没有任何 Team 操作记录。
7. 两个母号各自显示当前子号、下一子号、已用完数量和累计接力次数。
8. 应用在任一工作流阶段退出后，恢复执行不会重复邀请、重复退出或提前计数。
9. 接力成功后空间再次回到无 next 状态；下一次点击必须再次生成全新 Alias，不能复用远端旧 Alias。

## 7. 非目标与安全边界

- 本轮不自动清理现有 20+ 未接管旧 Alias。
- 本轮不在接力成功后自动停用或删除任何远端 Alias。
- 本轮不改变 Team 母号自身登录态，也不让母号参与接力工作流；母号只用于归属校验、默认子号链路和 HME 资源池生成配置。
- 不在用户未完成 iCloud、IMAP、S5 配置时发起检测、生成或删除。
- 不把 HME Cookie、IMAP 密码、代理凭据或远端匿名 ID写入计划、日志、截图或 API 响应。

### 节点 H：两个 LokiProxy 共用一个 Clash 前置（实现完成，等待供应商配额）

#### H.1 目标与性能指标

保留两个母号各自独立的 LokiProxy 动态源，同时让两条链都通过用户已经运行的同一个 Clash/Mihomo 本地代理建立外连。母号仍然只是被动归属锚点。

```text
工作流 A -> 本地 Team Workflow 中继 A -> 统一 Clash `127.0.0.1:7897` -> LokiProxy A -> 目标
工作流 B -> 本地 Team Workflow 中继 B -> 统一 Clash `127.0.0.1:7897` -> LokiProxy B -> 目标
```

- Team Workflow 只创建两个绑定 `127.0.0.1` 的轻量 SOCKS5 中继，不创建第二个 Clash 进程。
- 两个 LokiProxy 的生成请求和动态端点连接都使用同一个 Clash 前置 URL；不为 A/B 分配不同 Clash 节点或不同第一跳。
- 不写入 Mihomo 配置，不创建 Mihomo listener/provider/dialer-proxy，不调用 reload 或 selector 切换接口。
- 子号账号保存各自母号对应的本地中继地址；母号不建立 Team 网络会话。
- HME / IMAP 继续按资源池自己的直连设置运行，避免把 Apple Session 与 Team 子号出口混在一起。
- 动态 LokiProxy 端点只保存在进程内短期缓存，凭据不进入日志、前端响应或计划文件。

#### H.2 参考实现与采用结论

- 本机 Mihomo Meta `v1.19.21` 已确认存在 `mixed-port: 7897`，通过该端口访问 `icloud.com.cn` 和普通 HTTPS 均返回 `200`。
- GitHub 同类实现（`proxy.py` 的上游代理扩展、`singbox2proxy` 的 relay/chaining 模式）共同采用“本地入口与上游连接解耦”的结构；本项目只取其边界原则，不引入会接管系统代理的常驻代理进程。
- 采用标准 SOCKS5 CONNECT 中继：本地入口接收工作流目标，先经统一 Clash 建立到 LokiProxy 动态端点的连接，再由 LokiProxy 连接目标；生成 API 同样经统一 Clash 请求。
- Clash 选择器是唯一的第一跳控制面，Team Workflow 不复制、合并或修改其配置。

#### H.4 验收场景

1. Team Workflow 不创建自有 Clash 进程，不向现有 Mihomo 写入 listener/provider/dialer-proxy 或合并配置。
2. 工作流 A 的连接经过统一 Clash 后只到 LokiProxy A；工作流 B 的连接经过同一统一 Clash 后只到 LokiProxy B，二者端点和缓存互不串用。
3. 两条生成 API 请求的 `proxies` 地址完全相同，Clash 重启或用户切换出口后 Team Workflow 不覆盖或重置选择器。
4. 动态端点过期时只刷新对应母号；一个 LokiProxy 失败不污染另一条链，活动连接不因缓存刷新被强行断开。
