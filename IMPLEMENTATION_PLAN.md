# iCloud HME Team 接力实施计划

## 1. 目标与性能指标

把同一个 iCloud 账户抽象成一个 HME 资源池，在资源池内长期保护两个 Team 母号，并让每个 Team 空间只轮换一个子号。

闭环目标：

1. 从 `/v2/hme/list` 读取远端 Alias，只接管用户明确选择的 Alias。
2. 被标记为 `team_owner` 的母号永久保护，不进入账号池分配、退役或远端清理路径。
3. 被标记为 `rotating_child` 的子号进入现有账号池，并可作为空间当前号或下一号。
4. 每个 Team 母号分别保存自己的子号 S5；归属该母号的新子号默认继承这条 S5，之后仍可按账号覆盖。
5. 完整工作流成功后，下一号晋升为当前号；旧子号在同一个数据库事务中进入该母号的“已用完”池并永久保留计数。
6. 同一个 iCloud 资源池可以同时服务多个 Team 空间，每个空间绑定一个不同的母号。
7. Team 母号空间平时不预留下一号；用户点击“更换子号”时才现场创建新的 HME Alias、绑定为下一号并启动接力。

稳定性指标：

- 未显式选择的旧 Alias 零写入、零删除。
- `team_owner` 母号进入账号轮换或用完池的概率为零，由数据库约束和服务层双重阻断。
- 重复同步和重复导入均幂等。
- Apple HME 超时或 Session 过期不会阻塞已完成的 Team 轮换；接力完成后不自动修改远端 Alias。
- 敏感 Cookie、IMAP 密码、S5、远端 ID 继续加密保存且不进入日志或前端响应。

## 2. 已有系统分析

- 当前 `icloud_mailboxes` 已能保存一个 iCloud HME Session、转发 IMAP 和资源池 S5。
- 当前 `icloud_aliases` 仅记录本工具新建的 Alias，且每个 Alias 必须对应一个普通账号。
- 当前“检测”只返回远端 Alias 数量，不会同步已有 Alias。
- 当前八阶段工作流已经符合子号接力顺序：旧子号邀请新子号、旧子号退出、新子号登录并确认空间、创建令牌和推送。
- 当前成功提交会把旧账号标记为 `retired`，但不会清理对应的 iCloud Alias。
- 当前模型没有永久母号角色，也没有 Team 空间与母号的绑定关系。
- 当前成功提交会从普通可用账号中自动补下一号；iCloud 母号空间必须禁用这条预分配逻辑。

## 3. GitHub 同类项目经验

参考项目：

- `ultra2207/Accounts-Generator-public`：以 `/v2/hme/list` 为事实来源，对 `isActive` 和远端缺失状态做幂等同步。
- `not-knope/Hide-My-Email-iCloud-Manager`：远端清理需要先停用再删除，因此不适合作为接力成功事务的同步依赖。
- `heartmore/icloud-hme`：将 Alias 列表、生成、保留、停用和删除封装为独立 HME 生命周期操作。

采用的结论：

- 远端列表是测量反馈，不凭本地状态猜测 Apple 当前状态。
- 同步与接管分离：同步只展示，接管必须由用户选择。
- 接力成功只更新本地“已用完”状态，不自动停用或删除远端 Alias。
- 本地保留完整账号、母号归属和运行关联，避免审计链断裂并支持累计计数。

## 4. 数据模型设计

数据库升级到 schema v6：

- 重建 `icloud_aliases`，增加角色 `team_owner | rotating_child`，并允许母号没有账号池记录。
- 子号增加 `parent_owner_alias_id`，明确归属两个母号中的一个。
- 母号增加独立加密 S5；iCloud 资源池 S5 只负责 HME 与 IMAP，母号 S5 负责其子号的 ChatGPT 工作流。
- 子号增加 `used_at`，接力成功后进入“已用完”池但不删除本地或远端记录。
- `rotating_child` 必须有 `account_id`；`team_owner` 必须没有 `account_id`。
- `workspaces` 增加可空的 `owner_alias_id`，同一母号最多绑定一个空间。
- 现有 v5 Alias 全部无损迁移为 `rotating_child`。

关键约束：

- 母号不能变成可分配账号，不能被状态切换接口退役，不能进入清理队列。
- 空间绑定的母号必须是活动的 `team_owner`。
- 使用 iCloud 子号的空间，其母号和子号必须来自同一个 `icloud_mailbox`。
- 已用完子号不能重新进入可用账号候选，但继续参与母号和空间的累计统计。

## 6. 验收场景

1. 一个 iCloud 返回 20+ Alias，页面默认全部忽略；用户只选择两个母号和两个当前子号后，本地只新增四条受管记录。
2. 两个母号分别绑定两个空间，二者均不进入普通账号池。
3. 两个当前子号分别绑定为对应空间当前账号，且可各自覆盖独立 S5。
4. 空间平时没有下一号；点击“更换子号”后才创建一个全新 Alias，继承该母号 S5，并立即作为下一子号进入现有八阶段工作流。
5. 工作流失败或取消时，旧子号不进入已用完池；已生成的新子号和 next 绑定保留以便恢复或重试。
6. 工作流完整成功并提交轮换后，旧子号进入对应母号的“已用完”池；母号和远端 Alias 保持不变。
7. 两个母号各自显示当前子号、下一子号、已用完数量和累计接力次数。
8. 应用在任一工作流阶段退出后，恢复执行不会重复邀请、重复退出或提前计数。
9. 接力成功后空间再次回到无 next 状态；下一次点击必须再次生成全新 Alias，不能复用远端旧 Alias。

## 7. 非目标与安全边界

- 本轮不自动清理现有 20+ 未接管旧 Alias。
- 本轮不在接力成功后自动停用或删除任何远端 Alias。
- 本轮不改变 Team 母号自身登录态，也不让母号参与接力工作流。
- 不在用户未完成 iCloud、IMAP、S5 配置时发起检测、生成或删除。
- 不把 HME Cookie、IMAP 密码、代理凭据或远端匿名 ID写入计划、日志、截图或 API 响应。
