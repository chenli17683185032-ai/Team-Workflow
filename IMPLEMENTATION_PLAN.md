# iCloud HME Team 接力与应急提拉实施计划

## 1. 目标与性能指标

把同一个 iCloud 账户抽象成一个 HME 资源池，在资源池内长期保护两个 Team 母号，并让每个 Team 空间只轮换一个子号。母号在正常换班中是被动归属锚点，所有日常 Team 动作都由当前子号和下一子号完成；只有当前子号故障且用户明确点击“提拉”时，母号才进入独立应急状态机。

闭环目标：

1. 从 `/v2/hme/list` 读取远端 Alias，只接管用户明确选择的 Alias。
2. 被标记为 `team_owner` 的母号永久保护，不进入账号池分配、退役或远端清理路径。
3. 被标记为 `rotating_child` 的子号进入现有账号池，并可作为空间当前号或下一号。
4. 每个 Team 母号分别保存“该母号名下子号的默认 S5/动态链路”；归属该母号的新子号默认继承这条链路，正常换班不使用母号身份，提拉时母号使用同组链路和独立加密浏览器身份。
5. 完整工作流成功后，下一号晋升为当前号；旧子号在同一个数据库事务中进入该母号的“已用完”池并永久保留计数。
6. 同一个 iCloud 资源池可以同时服务多个 Team 空间，每个空间绑定一个不同的母号。
7. Team 母号空间平时不预留下一号；用户点击“更换子号”时才现场创建新的 HME Alias、绑定为下一号并启动接力。
8. Team 行保留无业务副作用的“编辑配置”，用于代理故障时修正该组子号默认链路；该入口不得登录 Team 或改变成员。
9. “提拉”逐人清退并读取远端反馈，确认只剩母号 1 人后才邀请新号，最终必须恰好恢复为母号与新子号 2 人。

稳定性指标：

- 未显式选择的旧 Alias 零写入、零删除。
- `team_owner` 母号进入账号轮换或用完池的概率为零，由数据库约束和服务层双重阻断。
- 重复同步和重复导入均幂等。
- Apple HME 超时或 Session 过期不会阻塞已完成的 Team 轮换；接力完成后不自动修改远端 Alias。
- 敏感 Cookie、IMAP 密码、S5、远端 ID 继续加密保存且不进入日志或前端响应。

## 2. 已有系统分析

- 当前 `icloud_mailboxes` 已能保存一个 iCloud HME Session、转发 IMAP 和资源池 S5。
- 当前 `icloud_aliases` 仅记录本工具新建的 Alias，且每个 Alias 必须对应一个普通账号。
- 当前“检测”只返回远端 Alias 数量，不会同步已有 Alias。
- 当前八阶段工作流已经符合子号接力顺序：旧子号邀请新子号、旧子号退出、新子号登录并确认空间、创建令牌和推送。
- 当前成功提交会把旧账号标记为 `retired`，但不会清理对应的 iCloud Alias。
- 当前模型没有永久母号角色，也没有 Team 空间与母号的绑定关系；母号不能作为工作流执行账号。
- 当前成功提交会从普通可用账号中自动补下一号；iCloud 母号空间必须禁用这条预分配逻辑。

## 3. GitHub 同类项目经验

参考项目：

- `ultra2207/Accounts-Generator-public`：以 `/v2/hme/list` 为事实来源，对 `isActive` 和远端缺失状态做幂等同步。
- `not-knope/Hide-My-Email-iCloud-Manager`：远端清理需要先停用再删除，因此不适合作为接力成功事务的同步依赖。
- `heartmore/icloud-hme`：将 Alias 列表、生成、保留、停用和删除封装为独立 HME 生命周期操作。

采用的结论：

- 远端列表是测量反馈，不凭本地状态猜测 Apple 当前状态。
- 同步与接管分离：同步只展示，接管必须由用户选择。
- 接力成功只更新本地“已用完”状态，不自动停用或删除远端 Alias。
- 本地保留完整账号、母号归属和运行关联，避免审计链断裂并支持累计计数。

## 4. 数据模型设计

数据库先升级到 schema v6 建立 iCloud 归属模型，再升级到 schema v7 区分正常换班与应急提拉：

- 重建 `icloud_aliases`，增加角色 `team_owner | rotating_child`，并允许母号没有账号池记录。
- 子号增加 `parent_owner_alias_id`，明确归属两个母号中的一个。
- 母号增加独立加密的“子号默认链路”；iCloud 资源池 S5 只负责 HME 与 IMAP，该默认链路只注入其名下子号的 ChatGPT 工作流。
- 子号增加 `used_at`，接力成功后进入“已用完”池但不删除本地或远端记录。
- `rotating_child` 必须有 `account_id`；`team_owner` 必须没有 `account_id`。
- `workspaces` 增加可空的 `owner_alias_id`，同一母号最多绑定一个空间。
- 现有 v5 Alias 全部无损迁移为 `rotating_child`。
- `runs.kind` 增加 `handoff | rescue`，旧运行自动迁移为 `handoff`。
- 母号的 SID、地域和浏览器指纹使用独立用途密文保存，不复用普通账号池身份。

关键约束：

- 母号不能变成可分配账号，不能被普通状态接口启停或退役，不能进入账号清理队列；Team 写操作只允许从提拉队列入口进入。
- 空间绑定的母号必须是活动的 `team_owner`。
- 使用 iCloud 子号的空间，其母号和子号必须来自同一个 `icloud_mailbox`。
- 已用完子号不能重新进入可用账号候选，但继续参与母号和空间的累计统计。

## 6. 验收场景

1. 一个 iCloud 返回 20+ Alias，页面默认全部忽略；用户只选择两个母号和两个当前子号后，本地只新增四条受管记录。
2. 两个母号分别绑定两个空间，二者均不进入普通账号池或正常换班阶段；只有显式提拉可以让对应母号登录、清退和邀请。
3. 两个当前子号分别绑定为对应空间当前账号；运行输入只包含当前子号和下一子号，且子号可各自覆盖默认链路。
4. 空间平时没有下一号；点击“更换子号”后才创建一个全新 Alias，继承该母号名下子号默认链路，并立即作为下一子号进入现有八阶段工作流。
5. 工作流失败或取消时，旧子号不进入已用完池；已生成的新子号和 next 绑定保留以便恢复或重试。
6. 正常换班完整成功并提交轮换后，旧子号进入对应母号的“已用完”池；母号和远端 Alias 保持不变，母号没有正常换班操作记录。
7. 两个母号各自显示当前子号、下一子号、已用完数量和累计接力次数。
8. 应用在任一工作流阶段退出后，恢复执行不会重复邀请、重复退出或提前计数。
9. 接力成功后空间再次回到无 next 状态；下一次点击必须再次生成全新 Alias，不能复用远端旧 Alias。

## 7. 非目标与安全边界

- 本轮不自动清理现有 20+ 未接管旧 Alias。
- 本轮不在接力成功后自动停用或删除任何远端 Alias。
- 正常换班不改变 Team 母号登录态；母号只在用户显式确认的提拉运行中登录和恢复成员，不向其他入口扩散该能力。
- 不在用户未完成 iCloud、IMAP、S5 配置时发起检测、生成或删除。
- 不把 HME Cookie、IMAP 密码、代理凭据或远端匿名 ID写入计划、日志、截图或 API 响应。

### 节点 H：两个 LokiProxy 共用一个 Clash 前置（实现完成，等待供应商配额）

#### H.1 目标与性能指标

保留两个母号各自独立的 LokiProxy 动态源，同时让两条链都通过用户已经运行的同一个 Clash/Mihomo 本地代理建立外连。母号仍然只是被动归属锚点。

```text
工作流 A -> 本地 Team Workflow 中继 A -> 统一 Clash `127.0.0.1:7897` -> LokiProxy A -> 目标
工作流 B -> 本地 Team Workflow 中继 B -> 统一 Clash `127.0.0.1:7897` -> LokiProxy B -> 目标
```

- Team Workflow 只创建两个绑定 `127.0.0.1` 的轻量 SOCKS5 中继，不创建第二个 Clash 进程。
- 两个 LokiProxy 的生成请求和动态端点连接都使用同一个 Clash 前置 URL；不为 A/B 分配不同 Clash 节点或不同第一跳。
- 不写入 Mihomo 配置，不创建 Mihomo listener/provider/dialer-proxy，不调用 reload 或 selector 切换接口。
- 子号账号保存各自母号对应的本地中继地址；母号不建立 Team 网络会话。
- HME / IMAP 继续按资源池自己的直连设置运行，避免把 Apple Session 与 Team 子号出口混在一起。
- 动态 LokiProxy 端点只保存在进程内短期缓存，凭据不进入日志、前端响应或计划文件。

#### H.2 参考实现与采用结论

- 本机 Mihomo Meta `v1.19.21` 已确认存在 `mixed-port: 7897`，通过该端口访问 `icloud.com.cn` 和普通 HTTPS 均返回 `200`。
- GitHub 同类实现（`proxy.py` 的上游代理扩展、`singbox2proxy` 的 relay/chaining 模式）共同采用“本地入口与上游连接解耦”的结构；本项目只取其边界原则，不引入会接管系统代理的常驻代理进程。
- 采用标准 SOCKS5 CONNECT 中继：本地入口接收工作流目标，先经统一 Clash 建立到 LokiProxy 动态端点的连接，再由 LokiProxy 连接目标；生成 API 同样经统一 Clash 请求。
- Clash 选择器是唯一的第一跳控制面，Team Workflow 不复制、合并或修改其配置。

#### I.1 目标与性能指标

#### I.2 GitHub 同类项目与采用结论

- `heartmore/icloud-hme`：把 HME API 客户端与网页登录/Cookie 导入分离；其多账号模型说明会话必须按资源池隔离，但不适合直接保存浏览器 Cookie 文件。
- `not-knope/Hide-My-Email-iCloud-Manager`：以远端 Alias 列表为事实来源，支持手动 Cookie 导入；远端管理逻辑可复用，但不能代替用户完成 Apple 登录。
- Playwright 官方网络事件模式：在 `BrowserContext`/`Page` 监听 response，先按 host/path 过滤，再读取 request URL 和允许的 headers；不记录全量 HAR。

采用结论：新增独立 `icloud_hme_capture.py` 捕获管理器，使用 Playwright 可见浏览器和临时 profile；`icloud_hme.py` 增加经过同一白名单校验的“请求字段 -> Session”解析函数；控制器通过状态 API 与现有资源池更新事务连接，避免把浏览器自动化代码塞进 HME HTTP 客户端。

#### I.3 最小充分模型

```text
用户登录/进入 HME 页面
```

### 节点 K：PAT + Session 自动导出 Sub2API JSON（完成）

#### K.1 目标与性能指标

- Session 只提供邮箱、Team account ID、user ID 和套餐等身份元数据；不得把 session token 写进 Sub2API 文件。

- 断点恢复不得重复创建 PAT；已有合法导出文件直接复用，文件丢失时仅从加密断点重建。
- Sub2API 管理员推送继续保持可选；本地导出不依赖服务地址、管理员邮箱、密码、TOTP 或 API Key。

### 节点 L：Team 活跃成员两人硬上限（已完成）

#### L.1 目标与性能指标

把“一个 Team 任意时刻不得存在超过两名活跃成员”提升为工作流安全不变量。正常接力固定为母号与当前子号两名活跃成员；新子号只保持待接受邀请，必须在旧子号退出成功后才允许登录并加入。

- 邀请前读取远端成员事实，`total > 2` 时在任何写操作前中止。
- 邀请列表最多只允许本次目标子号；存在其他待接受邀请时中止，避免第三方并发入组。
- 旧子号退出接口未成功返回时，绝不进入新子号登录阶段；接口成功后最长等待 15 秒成员反馈，旧号仍存在或旧号缺席但已有两名无关成员时同样不登录新号。

- 断点恢复仍必须经过最终成员复核；旧版本已存在的 PAT 断点也不能绕过复核。

### 节点 N：iCloud Team Workspace 自动识别（完成）

#### N.1 现状与目标

导入 iCloud Team 时，前端把 `Workspace ID` 作为人工必填项；本地其实已经具备当前子号 OTP 登录、Session 交换和成员只读接口，却没有把这些测量组合成导入前的闭环。结果是母号与子号都已选中时，提交按钮仍因未知 Workspace ID 被锁死。

目标是把 Workspace ID 从人工配置改为系统测量值：当前子号只读登录后枚举可访问 Team，对每个候选读取成员，只接受“所选母号与当前子号同时存在、成员总数恰好为 2、列表完整”的唯一候选。识别成功后才能继续本地导入；零匹配、多匹配、超员、成员列表不完整或网络身份不一致均停止，不猜测、不复用其他组 ID。

性能与安全指标：

- 正常导入只需选择母号、当前子号和该组代理；提交时自动识别 Workspace ID 并继续原有幂等导入。
- 识别只允许登录、Session 刷新和成员读取，不调用邀请、退出、删除成员、创建 Alias、创建 PAT 或账号注册。
- 当前子号登录继续使用 `本机 Clash -> 该组 LokiProxy -> OpenAI`；IMAP 只使用本地资源池当前配置，不使用项目历史或作者遗留凭据。
- 远端必须恰好 2 名活跃成员且身份完整；任何不确定性均在本地导入前失败。
- Session、Access Token、代理凭据、Workspace 候选和成员明细不落库、不进入 API 响应、普通日志或 DOM；响应只返回最终唯一 Workspace ID 与人数校验摘要。
- 已存在且身份一致的本地 Team 直接返回既有 Workspace ID，不重复登录；并发 iCloud 写操作或识别操作保持串行。

### 节点 P：通用 Clash 两跳代理导入（完成）

#### P.1 目标与性能指标

把当前供应商特定的 “LokiProxy 模式” 综合为通用 Team 子号代理链。用户在前端直接粘贴 CliProxy 等渠道提供的完整 HTTP 代理 URL，系统自动把它作为第二跳，第一跳默认且强制使用本机共享 Clash：

```text
Team 工作流 -> 本地隔离中继 -> 共享 Clash -> CliProxy/其他固定代理 -> 目标
```

性能、稳定性与安全指标：

- 新建/导入 Team 和“编辑配置”默认选择 `Clash 两跳`；粘贴完整 `http://[user:password@]host:port` 即可保存，不要求选择 Loki 供应商或伪装成 SOCKS5。
- Team 的固定 HTTP、固定 SOCKS5 和历史 Loki 生成链接都使用同一个 Clash 第一跳；前端不能为不同 Team 指定不同第一跳，也不能在默认流程中绕过它。
- 旧 `lokiproxy_generator` 密文配置无损读取，并在下一次保存时规范化为供应商无关的 `clash_chain`；现有中继端口、账号绑定和运行身份保持不变。
- HTTP URL 的用户名、密码和完整源地址只进入 Keychain/AES-GCM 密文；API、DOM、普通日志、错误消息和计划文档不回显。
- 固定 HTTP/SOCKS URL 不发起“生成器请求”；只有合法 Loki `/gen` 历史链接才按动态源获取，避免把 CliProxy 端点误当网页 GET。
- LaunchAgent 停机小于 1 分钟；部署后现有组 1 配置仍为 `PH / Asia/Manila`，不触发新的 Alias、Team 成员或换班写操作。

#### P.2 GitHub 与既有经验

- GitHub 未检索到可信的 CliProxy 专用开源协议实现；CliProxy 本次提供的是标准 HTTP CONNECT 代理，按协议能力建模，不按供应商品牌硬编码。
- `MetaCubeX/Meta-Docs` 与公开 Mihomo `dialer-proxy` 示例证明“第一跳拨号代理 -> 第二跳出口代理”是通用链式结构。本项目继续使用已验证的用户态中继，不写入 Mihomo 配置、不调用 reload 或 selector API。
- 当前真实闭环已证明 `Clash -> 固定 HTTP CONNECT -> ipwho.is` 可稳定返回菲律宾精确时区；该最小充分模型先固化，再兼容固定 SOCKS5和历史生成链接。

### 节点 Q：iCloud 登录浏览器长期会话记忆（完成）

#### Q.1 现状、目标与性能指标

当前“登录更新 HME”每次都为 Chrome for Testing 创建临时 profile，捕获结束后无条件删除。浏览器因此无法在两次捕获间保留 Cookie、Apple 设备信任、站点存储和本地密码管理数据，导致用户反复输入密码和双重验证。

目标是为每个 iCloud 资源池提供独立、长期保留的浏览器 profile：同一资源池后续捕获复用同一登录状态，不同资源池绝不共享 Cookie。捕获结束仍关闭浏览器进程，但不再删除持久 profile。

性能、稳定性与安全指标：

- 同一邮箱 ID 无论服务重启多少次都映射到同一目录；不同 ID 映射到不同目录。
- profile 放在 Git 工作区外的应用数据目录，邮箱 ID 只以稳定哈希出现在路径中，目录权限固定为 `0700`。
- 不读取或锁定用户日常 Chrome profile；继续使用项目独立的 Chrome for Testing/Chromium。
- Cookie、密码库、Local Storage、缓存和 Apple 信任状态不进入 SQLite、API、DOM、日志、测试产物或 Git。
- Apple 服务端主动过期、撤销 Session 或再次要求 2FA 时仍以真实反馈为准，不伪造或绕过验证。

### 节点 R：Sentinel 预取超时的总时限改造（完成）

#### R.1 现状、测量与目标

`create_account` 在 `check_proxy` 后异步预取浏览器 Sentinel，与邮箱 OTP 并行。现有捕获器中，`page.goto(wait_until="domcontentloaded")`、备用 `page.goto(wait_until="commit")` 和 `page.wait_for_function(SentinelSDK)` 各自使用独立的 timeout，因此一次不可用可以串联消耗超过两分钟。

2026-07-19 实际运行证据：预取从 `21:53:10` 开始，`21:54:14` 以 `wait_for_function` 30 秒超时失败；同步重试到 `21:56:18` 才成功取到 token，单个账户创建因此多出约 124 秒。

目标是把一次 Sentinel 捕获的所有阶段收敛在一个总时限内：

- 第一次导航超时后，备用导航只能消耗剩余时间，不再重新获得一个完整 timeout。
- `wait_for_function` 无法取到 `SentinelSDK` 时仍保持原有错误和关闭浏览器行为，但不会超过本次捕获总预算。
- 预取成功时不改变 token、flow、指纹、代理或 OpenAI 请求头；只改超时分配。
- 失败重试不会绕过 Sentinel 校验，也不会在本地伪造 token；仍由浏览器和远端返回决定是否继续。

#### R.2 GitHub 与既有实现经验

- `microsoft/playwright-python` 与 `microsoft/playwright` 的 `goto` / `wait_for_function` API 均把 timeout 作为每个调用的独立选项，并不跨调用扣减剩余时间。本节点采用其上层调用者维护 deadline 的做法。
- 现有 `sentinel_browser.py` 已有统一资源关闭和代理 lease 回收；保留这些边界，只引入一个内部剩余时间计算器。
- 当前 `register.py` 已经把预取与 OTP 并行，故先修复单次捕获的时间放大，不重构邮件提供方、队列或浏览器指纹。

#### R.3 最小充分闭环

```text
进入 Sentinel 捕获
    -> deadline = now + timeout_seconds
    -> 首次导航使用 remaining(deadline)
    -> 如果需要，备用 commit 导航只用剩余预算
    -> warmup 和 SentinelSDK 等待同样消耗该预算
    -> 剩余为 0 时立即失败并清理浏览器
    -> 预取失败时只执行一次不超过总预算的合法重试
```

控制对象是单次 Sentinel 捕获的稳定时间，测量是导航响应、SDK 存在性和 token 返回，执行器是 Playwright 页面。以总 deadline 作负反馈，把代理时延、页面资源不完整和 SDK 未加载视为扰动，优先保证不超时串联。
